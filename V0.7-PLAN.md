# Knit v0.7 — design plan

**Status:** ✅ v0.7.0 shipped (steps 1–7 + audit fixes); ✅ v0.7.1 shipped (`tools/list_changed` hot-reload); ✅ v0.7.2 shipped (token budget guardrail, update notification, integration scanner, README quiet-mode). **Theme:** Knit becomes the *connective tissue*, not another framework. Universal applicability, dynamic per-project tailoring, honest token discipline.

## The four pillars

### 1. Server-level `instructions` (universal protocol injection)

MCP servers can set an `instructions` field on the handshake. Claude Code (and any MCP client — Cursor, Codex, etc) injects this into the session's system prompt automatically, *before* tool descriptions are read.

Currently `src/mcp/server.ts:29` initializes without it. Adding it closes the "agent doesn't know the flow at session start" gap.

**Universal baseline (~400-500 tokens)** — always injected:

```text
Knit is a memory + workflow layer for this project.

ALWAYS at session start:
1. knit_load_session — returns prior handoff, learnings, false positives.
   If has_unfinished_work: true, resume that handoff.
2. For non-trivial work, knit_classify_task before any Edit/Write —
   returns tier and phases.
3. If tier=complex with auto_plan_mode=true, EnterPlanMode immediately.
4. Before reporting done, knit_record_learning if anything non-obvious
   surfaced (not a substring match on prior learnings).

When to use other tools:
- knit_query_imports/exports/dependents/tests — instead of grep when index is fresh
- knit_search_learnings — before re-investigating a domain
- knit_search_sessions — "have I done this before?"
- knit_get_workflow — fetch phase depth on demand; do not try to remember it

The protocol enforces a 4-tier classifier (Inquiry / Trivial / Standard / Complex).
Knit provides inputs; you make the calls.
```

**Per-project tailoring (added based on integration scan, ~100-300 tokens):**

```text
DETECTED: user has gstack installed.
  Use /plan, /ship, /qa from gstack for routing.
  Knit provides only memory + classification + retrieval.

DETECTED: existing memory tool X.
  Compatibility mode — Knit writes only what X doesn't cover.
```

### 2. Tier-gated tool registry (LOCKED design)

The MCP `tools/list` response is computed per session. Server returns only the subset that fits this project shape. Three tiers.

#### Tier 1 — Core (always exposed, 24 tools)

Universal — every project type benefits regardless of stack.

**Memory + retrieval (8)**
- `knit_load_session`
- `knit_search_learnings`
- `knit_search_global_learnings`
- `knit_search_sessions`
- `knit_record_learning`
- `knit_record_global_learning`
- `knit_save_session_summary`
- `knit_save_handoff`

**Knowledge graph (5)** — Knit's defining differentiator, always available; returns honest empty result on non-analyzable projects
- `knit_query_imports`
- `knit_query_exports`
- `knit_query_dependents`
- `knit_query_tests`
- `knit_find_fanout`

**Workflow + classification (4)**
- `knit_classify_task`
- `knit_build_context`
- `knit_get_workflow`
- `knit_get_suggestions`

**False positives + reflection (3)** — universal; FP machinery is core to reviewer-agent workflow
- `knit_record_false_positive`
- `knit_get_false_positives`
- `knit_reflect`

**Protocol Guard config (2)** — universal; every install ships Protocol Guard
- `knit_set_protocol_strictness`
- `knit_get_protocol_strictness`

**Diagnostics + meta (2)**
- `knit_brain_status`
- `knit_list_features` *(NEW — meta-tool for tier discoverability)*

#### Tier 2 — Conditional (auto-exposed when detected, 10 tools)

**Team worktrees (9)** — only when ≥3 domains detected OR `knit_enable_feature("teams")`. Solo-domain projects don't see 9 cousins cluttering their decision space.
- `knit_spawn_team_worktree`
- `knit_finalize_team_worktree`
- `knit_list_team_worktrees`
- `knit_define_team`
- `knit_get_teams`
- `knit_get_team_prompt`
- `knit_start_team_review`
- `knit_post_team_findings`
- `knit_get_board_summary`

**Subagents (1)** — only when `.claude/agents/` exists OR user enables
- `knit_install_agent`

#### Tier 3 — Admin / one-time (hidden by default, 2 tools)

- `knit_prune_sessions` — admin; reachable via `knit_enable_feature("admin")`
- `knit_setup_project` — hidden after first run; only useful for initial bootstrap

**Count check:** 24 (T1) + 10 (T2) + 2 (T3) = 36 tools (35 existing + `knit_list_features` new).

#### `knit_list_features` — the discoverability escape hatch

Always in Tier 1. Returns:

```json
{
  "active": [...tools currently exposed],
  "available": [...tools not exposed but enable-able],
  "reasoning": {
    "knit_spawn_team_worktree":
      "Hidden — project has 1 detected domain.
       Enable via knit_enable_feature('teams')."
  }
}
```

The agent calls this when the user asks "can I do X" and the tool isn't visible — instead of saying "Knit doesn't support that," the agent surfaces "yes, but it's not active by default for this project shape — want me to enable it?"

### 3. Inquiry tier in the classifier (real bug fix)

The workflow protocol documents 4 tiers (Inquiry / Trivial / Standard / Complex). The classifier in `src/mcp/handlers.ts:191` only returns 3 (drops Inquiry). This caused the v0.6.3 ship-readiness session to over-route an audit task into plan mode.

**Fix:** add intent detection before file/domain counting.

```text
Description matches /^(audit|explain|what|where|status|list|summarize|review|check)\b/i
  → tier=inquiry, phases=[]
  → instruction: "Read-only task. Answer directly. No plan mode, no LEARN
                   unless something durable surfaced."

Otherwise → fall through to existing file/domain count logic.
```

~30 LOC. Would have saved ~30k tokens of plan-mode ceremony in the v0.6.3 session alone.

### 4. Vectorless RAG (BM25 + import-graph fusion)

Replace the current substring-match search with BM25 over learnings + sessions + global pool, fused via Reciprocal Rank Fusion (k=60) with a knowledge-graph traversal retriever.

**Why vectorless:**
- Privacy — learnings never leave disk
- Zero cold-start — no model download
- Determinism — same query → same results
- Footprint — ~10kb gzipped vs ~10mb local embeddings

**Implementation (~300 LOC, zero deps):**
- BM25 indexer over learnings + sessions + global
- Knowledge-graph retriever: if a query matches a learning mentioning `auth.ts`, walk one hop and surface learnings about files that import `auth.ts`
- RRF fuses the two rankings
- Porter stemming (~50 LOC) closes morphological gap (`auth` ≈ `authentication` ≈ `login`)

**Hit-rate target:** R@5 ≥ 90% (agentmemory benchmark suggests 95% from BM25 + graph fusion vs 86% from BM25 alone).

## Token optimization (the discipline)

### Budget table

| Surface | Today (v0.6.5) | v0.7 target | Mechanism |
|---|---|---|---|
| Tool registry (typical project) | 6-8k | 3-4k | Tier gating + description compression |
| Server instructions | 0 | ~500 | NEW |
| CLAUDE.md (per-turn) | ~16.7KB | ~6KB | Generator trim + lazy phase content |
| `knit_load_session` response | 3-5KB | 1.5KB | Aggressive truncation, lazy load |
| `knit_classify_task` response | ~500 tok | ~150 tok | Drop `reasoning` by default |
| `knit_get_workflow` (per phase) | 1-3KB | 0.5-1.5KB | Trim ceremony, keep decision aids |
| **Per-turn fixed cost** | **~30-35k tok** | **~12-15k tok** | All of above |

Net savings per power-user session (50 turns): ~750k-1M tokens. At Sonnet 4.6 rates, that's ~$10-15 per heavy session.

### Mechanisms

1. **Tier gating** — locked above. Solo-domain projects drop 9 team tools immediately.

2. **Tool description compression** — current descriptions average ~150 chars; trim to ~60 chars one-liners. Example:
   - Before: `"Opt-in. Record a learning to the cross-project pool when the insight generalizes beyond this project (e.g., Stripe webhook signature rules). Per-project knit_record_learning stays primary."`
   - After: `"Record cross-project learning. Generalizable insights only."`
   - 70% reduction. Multiply across 36 tools.

3. **CLAUDE.md trim** — `src/generators/claude-md.ts` outputs 16.7KB today. Targets:
   - Drop the auto-generated full project map (high-fanout + untested lists capped at 5 each, not full)
   - Drop the boilerplate "system-reminder override" paragraph (covered by server instructions now)
   - Drop the phase-status placeholder section (template noise)
   - Net target: 6KB

4. **`knit_load_session` lazy fields** — return small core (last_session 200 char, handoff 1.5KB, top-3 learnings) by default; opt-in `include=patterns,teams,metrics` query parameter for the rest.

5. **`knit_classify_task` minimal mode** — return only `{tier, phases, instruction}` by default. The current `reasoning` and `cross_domain_ripple` fields move to `verbose=true` opt-in. Saves ~350 tokens per non-trivial task.

6. **`knit_get_workflow` content trim** — each phase string in `workflow-protocol.ts` is ~3KB; aim for ~1-1.5KB. Keep decision aids and tier vocabulary; cut ceremonial narrative ("READ → GREEN → REFACTOR expanded", handoff prose, etc).

7. **`knit_list_features` instead of always-exposed verbose tool list** — agent discovers what's available via metadata, not by reading 36 tool definitions upfront.

8. **Hook output discipline** — Stop hooks currently print ~100 tokens of LEARN-compliance reminder. Move to JSON-only output that Claude Code surfaces as a compact prefix rather than a 4-line console block.

### Token guardrail in the LEARN protocol

Add a budget check to `knit_brain_status`:

```text
"token_accounting": {
  "claude_md_bytes": 6100,        // target ≤ 6500
  "tool_registry_estimate": 3200, // target ≤ 4000
  "per_session_overhead": 9300,   // target ≤ 12000
  "verdict": "healthy" | "warn" | "over budget"
}
```

If any project drifts over budget, `knit_brain_status` surfaces it. The discipline becomes measurable, not vibes-based.

## What this enables

- **MCP for anything** becomes literal. Docs repo, Obsidian vault, ML notebook, Rust CLI — each sees a focused, project-shaped tool surface.
- **Workflow merging, not overlapping.** Users with existing frameworks (gstack, custom CLAUDE.md) keep their flow; Knit fills gaps.
- **Per-session token cost cut ~60%.** Honest, measurable, in `knit_brain_status`.
- **Discoverability without clutter.** `knit_list_features` surfaces hidden tools when relevant, no need to expose everything upfront.
- **Retrieval that actually retrieves.** BM25 + graph fusion ≈ 90%+ R@5 vs 86% from BM25 alone vs ~50-60% from current substring.
